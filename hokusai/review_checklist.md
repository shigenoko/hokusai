# レビューチェックリスト

このチェックリストは、過去のPRレビューで指摘された事項を基に作成されています。
最終レビュー（Phase 7）で自動的にチェックされます。

---

## 衛生管理・コード品質 (HQ)

### HQ01: デッドコードと不要なインポート
- [ ] 未使用の型定義がないか
- [ ] 未使用の変数・フィールドがないか
- [ ] 未使用のインポートがないか
- [ ] 使用しているコンポーネント・関数が全てインポートされているか
- [ ] コメントアウトされたコードが残っていないか
- [ ] デバッグ用の `console.log` が残っていないか（本番用ログは除く）

### HQ02: 重複コードと車輪の再発明防止
- [ ] 同じロジックが複数箇所に重複していないか
- [ ] フォールバックチェーン（`a ?? b ?? c`）が複数箇所で繰り返されていないか
- [ ] プロジェクト共通のユーティリティ・ライブラリで代替できる処理を独自実装していないか

### HQ03: 開発用ファイル・機密情報混入防止
- [ ] テストデータ生成スクリプトがコミットに含まれていないか
- [ ] 個人用の設定ファイル（`.personal/`等）が含まれていないか
- [ ] APIキー、パスワード、トークンがハードコードされていないか
- [ ] 環境変数で管理すべき値が直接書かれていないか

### HQ04: コードベースパターンの一貫性
- [ ] 新規コードが既存コードの慣習・パターンに従っているか
  - 例: エラーハンドリング方法、認証アクセス方法、API呼び出しパターン
- [ ] 同一機能の実装方法が既存コードと統一されているか（独自方式ではなく）
- [ ] 設定値・定数が既存のconfigや定数定義から取得されているか（ハードコードではなく）

### HQ05: 変更スコープの妥当性
- [ ] 変更された全ファイルがPR目的に直接関連しているか
- [ ] 無関係なリファクタリングやフォーマット変更が混入していないか

## セキュリティと認可 (SA)

### SA01: サーバーサイド認可
- [ ] APIエンドポイントで適切な権限チェックが行われているか
- [ ] リソースの所有者検証（UserID一致等）がサーバーサイドで実装されているか
- [ ] 管理者向けAPIと一般ユーザー向けAPIで認可レベルが適切に分離されているか

### SA02: フロントエンド制御とバックエンド認可の整合
- [ ] UIの非表示/無効化制御に対応するバックエンド認可チェックがあるか
- [ ] フロントエンドの制御だけに依存した認可になっていないか

---

## 型定義・設計 (TD)

### TD01: スキーマ形式の一貫性
- [ ] 新規追加した Request/Response スキーマが、同一コードベース内の**既存の類似スキーマと同じ形式**を使用しているか
  - 例: 既存が `discriminatedUnion("kind", [...])` なら、新規も同じパターンを使う
- [ ] 同じドメイン概念（背景、BGM、アイテム等）を表すフィールドが、全てのスキーマで統一された構造になっているか
- [ ] 4層（OpenAPI / Zod / Frontend / Server）で同じリソースの表現形式が一致しているか
- [ ] OpenAPI の `operationId` が backend 実装メソッド名と整合しているか
- [ ] ByAdmin 系の命名規則（`*ByAdmin`）が API / backend 間で一致しているか

### TD02: 型の再利用と戻り値型一貫性
- [ ] 同一構造の型定義が重複していないか（→ 統一してエイリアスにする）
- [ ] 既存の型を再利用できる場合、新規定義ではなく再利用しているか
- [ ] 必須であるべきパラメータがオプショナルになっていないか
- [ ] `SuccessOKResponse` を返す関数で `null` を成功値として返していないか
- [ ] 戻り値は `SuccessOKResponse | ErrorResponse` の二分に統一されているか

### TD03: 型安全な条件分岐・比較
- [ ] 混合型（string/number混在等）の比較で適切なフォールバックがあるか
- [ ] `null`/`undefined` チェックが適切か
- [ ] オプショナルチェーン（`?.`）の使用が適切か
- [ ] 呼び出し側で `status !== 200` による分岐を使用しているか（`"err" in` 等のプロパティ存在チェックではなく）

## データ処理と非同期 (DA)

### DA01: ソート・ページネーション・データ取得
- [ ] **ソートはページネーションの前に適用されているか**
  - 誤: `data → paginate → sort`（現在ページ内のみソート）
  - 正: `data → sort → paginate`（全データをソート後にページ分割）
- [ ] フィルター/ソート変更時にページ番号がリセットされるか
- [ ] `limit` を指定する場合、`orderBy` も指定しているか
- [ ] 大量データ取得時の警告・ページネーション対応があるか

### DA02: タイムゾーン
- [ ] 日付・時刻の処理で明示的なタイムゾーン指定があるか
- [ ] 「日本時間」と記載しているなら、JST(UTC+9)として明示的に変換しているか

### DA03: 非同期処理の安全性
- [ ] 独立した非同期処理を直列 `await` で逐次実行していないか（`Promise.all` で並列化可能か検討）
- [ ] 同一リソースへの並行アクセスで競合状態が発生しないか
- [ ] 複数ドキュメントの更新でトランザクションが必要な箇所にトランザクションが使われているか

### DA04: 境界条件の一貫性
- [ ] 日時比較で包含/排他（`<` vs `<=`）が既存ロジックと整合しているか
- [ ] ミリ秒精度が必要な比較で、精度の不整合がないか
- [ ] 範囲指定（開始〜終了）で、開始は包含・終了は排他というルールが統一されているか

## UI/UX・フロントエンド (UX)

### UX01: アクセシビリティと操作性
- [ ] リストのキーに意味のある識別子を使用しているか（インデックスだけでなく）
- [ ] ソート可能なリストで、行のキーが一意かつ安定しているか
- [ ] クリック可能な要素に `onKeyDown` ハンドラがあるか（Enter/Space対応）
- [ ] フォーカス可能であるべき要素に `tabIndex={0}` があるか
- [ ] フォームのバリデーションが適切か

### UX02: 画面遷移パターンとエラー表示
- [ ] 作成・編集フォームは別ページ（専用の編集ページ）で実装されているか（インラインフォームではなく、ページ遷移で行う）
- [ ] 編集ページは Card レイアウト（ヘッダーに閉じるボタン、フッターにキャンセル＋送信ボタン）を使用しているか
- [ ] エラー時にユーザーに適切なフィードバックがあるか
- [ ] ローディング状態が表示されるか

## カスタムルール

プロジェクト固有のルールは設定ファイル（`claude-workflow.yaml`）の
`review_checklist` セクションで追加できます。

```yaml
# 新形式（推奨）
review_checklist:
  P19:
    name: "stopLiveForDeletionの重複取得禁止"
    description: "stopLiveForDeletion 呼び出し前に getLive を事前取得しない（状態確認の責務は停止処理側に一元化）"
  P21:
    name: "ManageRequestのフィールド制限"
    description: "Manage*Request に updated_at など同時編集制御前提のフィールドを入れない"

# 旧形式（後方互換）
review_checklist:
  - "stopLiveForDeletion 呼び出し前に getLive を事前取得しない"
  - "Manage*Request に updated_at などのフィールドを入れない"
```
