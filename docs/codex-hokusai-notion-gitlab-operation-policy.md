# HOKUSAI 導入時の Notion / GitLab 運用方針

## 目的

HOKUSAI を導入先企業で活用する際に、Notion、GitLab Epic、GitLab Issue、GitLab MR をどのように扱うかを整理する。

本方針の目的は、情報の置き場所を明確にし、仕様・設計・進捗・コードレビューが二重管理にならない運用を作ることである。

## 結論

HOKUSAI の標準運用では、以下の分担を推奨する。

| 対象 | 使用判断 | 主な用途 | 正本として扱う情報 |
|---|---|---|---|
| Notion | 使う | 企画、要件、調査、設計、作業計画、意思決定ログ | 上位情報とドキュメント |
| GitLab Epic | 原則使わない | 使う場合は進捗集約・ロードマップ表示のみ | 原則なし |
| GitLab Issue | 使う | HOKUSAI が処理する実装タスク単位 | 実装タスクの状態 |
| GitLab MR | 使う | コード差分、レビュー、CI、マージ判断 | コード変更とレビュー結果 |

標準構成では、**Notion を企画・設計・判断の正本、GitLab を開発実行の正本**として扱う。

## 推奨する役割分担

### Notion

Notion は、HOKUSAI における上位情報と長文ドキュメントの正本として扱う。

主な用途:

- 企画の背景、目的、期待効果
- 業務要件、仕様、制約条件
- 調査レポート
- 設計書
- 実装計画
- 意思決定ログ
- 関連する GitLab Issue / MR の一覧
- 複数 Issue を束ねる親ページ

HOKUSAI の Phase 2 / Phase 3 / Phase 4 では、調査・設計・作業計画のような長文成果物を Notion 子ページとして保存する設計になっている。そのため、導入先企業でもドキュメント正本は Notion に寄せるのが自然である。

### GitLab Epic

GitLab Epic は、標準運用では原則使わない。

理由:

- Notion の親ページと役割が重複しやすい
- Epic に背景、要件、設計を書き始めると Notion と二重管理になる
- HOKUSAI の実行単位は Epic ではなく、より小さい Issue / MR 単位である

ただし、導入先企業が GitLab Roadmap や Epic Board を標準運用している場合は、GitLab Epic を補助的に使ってよい。

その場合の用途:

- 複数 GitLab Issue の集約
- 期限、マイルストーン、進捗率の可視化
- 経営層、PM、プロジェクト責任者向けのロードマップ表示
- 複数プロジェクトを横断した進捗確認

GitLab Epic を使う場合でも、Epic 本文には詳細な仕様や設計を書かない。Notion 親ページへのリンク、短い概要、関連 Issue の集約に留める。

### GitLab Issue

GitLab Issue は、HOKUSAI が処理する実装タスク単位として扱う。

主な用途:

- HOKUSAI に渡す実装タスク
- 担当者、ラベル、ステータスの管理
- 作業対象の短い説明
- Notion 親ページへのリンク
- 関連 MR へのリンク
- 短い進捗コメント

GitLab Issue には、長文の設計書や調査レポートを直接書き込まない。Issue は開発者が作業を開始し、状態を追跡し、MR に接続するための実行単位として扱う。

### GitLab MR

GitLab MR は、コード変更、レビュー、CI、マージ判断の正本として扱う。

主な用途:

- コード差分
- Draft MR の作成
- レビューコメント
- 修正対応
- CI 結果
- マージ可否の判断
- 最終的な変更履歴

MR には、Notion の設計書や実装計画へのリンクを載せる。MR 本文には、レビュアが判断できる範囲の要約、変更内容、検証結果を記載する。

## 情報の粒度別配置

| 情報の粒度 | 推奨配置 | 例 |
|---|---|---|
| 大きなテーマ | Notion | 「Slack 通知対応」「認証導線改善」「設定画面刷新」 |
| 要件・仕様 | Notion | 背景、目的、業務要件、制約 |
| 調査・設計 | Notion | 調査レポート、設計書、実装計画 |
| 実装タスク | GitLab Issue | 「Slack Webhook 設定を追加する」 |
| コード変更 | GitLab MR | Issue に対応する実装差分 |
| レビュー・CI | GitLab MR | レビューコメント、パイプライン結果 |
| 進捗集約 | Notion | 関連 Issue / MR の一覧、現在の状態 |

## 推奨するリンク構造

標準運用では、Notion を起点に GitLab Issue と MR を参照できる状態にする。

```text
Notion 親ページ
  - 背景
  - 目的
  - 要件
  - 調査結果
  - 設計書
  - 実装計画
  - 関連 GitLab Issues
      - Issue #101
      - Issue #102
  - 関連 GitLab MRs
      - MR !201
      - MR !202

GitLab Issue
  - HOKUSAI が処理する 1 タスク
  - Notion 親ページへのリンク
  - 実装対象
  - ステータス
  - 関連 MR リンク

GitLab MR
  - コード差分
  - レビューコメント
  - CI 結果
  - Notion 設計書へのリンク
  - 関連 Issue リンク
```

## GitLab Epic を使う場合の例外ルール

GitLab Epic を使う場合は、以下のルールを守る。

- Epic はロードマップや進捗集約のためだけに使う
- Epic 本文には詳細な仕様、設計、作業計画を書かない
- 詳細情報は Notion に置き、Epic から Notion へリンクする
- Epic 配下には GitLab Issue を紐づける
- MR は Issue に紐づけ、Epic へ直接詳細を集約しない

このルールにより、Notion と GitLab Epic の二重管理を避ける。

## 導入先企業への説明方針

導入先企業には、以下のように説明するのがよい。

> HOKUSAI では、Notion を企画・設計・判断の場所として使い、GitLab を実装・レビュー・CI の場所として使います。GitLab Issue は HOKUSAI が処理する実装タスク、GitLab MR はコードレビューとマージ判断に使います。GitLab Epic は標準では使わず、必要な場合のみロードマップや進捗集約のために限定的に使います。

## 最終判断

| 対象 | 標準判断 | 補足 |
|---|---|---|
| Notion | 使う | 上位情報、設計、調査、作業計画の正本 |
| GitLab Epic | 原則使わない | 使う場合は進捗集約・ロードマップ表示に限定 |
| GitLab Issue | 使う | HOKUSAI が処理する実装タスク単位 |
| GitLab MR | 使う | コード差分、レビュー、CI、マージ判断の正本 |
