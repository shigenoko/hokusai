# HOKUSAI導入に伴うツールチェイン（Notion / GitLab）運用指針

**対象読者:** プロジェクトマネージャー、リードエンジニア、開発責任者

本ドキュメントは、AI支援型開発ワークフロー「HOKUSAI」を導入するにあたり、既存のツールチェイン（Notion, GitLab）をどのように役割分担し、運用すべきかを整理したものです。

---

## 1. 基本方針：管理のNotion、実行のGitLab

HOKUSAIを最大限に活用し、チーム全体の生産性を向上させるための最重要方針は**「管理と意思決定はNotionに集約し、コードの実行と統合はGitLabに特化させる」**ことです。

情報の分散と二重管理を防ぐため、以下のツール選定を推奨します。

| ツール / 機能 | 採用方針 | 主な役割 |
| :--- | :---: | :--- |
| **Notion** (Epic / Task管理) | **必須（推奨）** | **「プロジェクトの司令塔」**<br>要件定義、タスク管理、AIによる設計・計画ドキュメントの蓄積。 |
| **GitLab Epic** | **不使用** | プロジェクト管理はNotionに一本化するため使用しません。 |
| **GitLab Issue** | **不使用** | 開発タスクはNotion上でチケット化するため使用しません。 |
| **GitLab MR** (Merge Request) | **必須** | **「作業現場・品質担保」**<br>コードの差分確認、自動テスト(CI/CD)、人間とAIのコードレビュー。 |

---

## 2. 各ツールの役割と運用詳細

### 2.1. Notion：情報の中心（The Brain）

HOKUSAIのワークフローにおいて、Notionは「タスクの起点」であり「情報の最終保存先」となります。

*   **Epic（プロジェクト・機能群）の管理**
    *   **運用:** 「ユーザー認証の刷新」のような大きな機能群は、Notionのデータベース（Epic DB）で管理します。ここに背景、ビジネス要件、デザインカンプ（Figma等）を集約します。GitLab Epicsは使用しません。
*   **Task（開発最小単位）の管理**
    *   **運用:** Epicから切り出された個別の開発タスクは、NotionのTask DBで管理します。これがHOKUSAIの実行起点（チケット）となります。GitLab Issueは使用しません。
*   **AI生成ドキュメントの蓄積**
    *   **運用:** HOKUSAIが稼働すると、Taskページ配下に「事前調査レポート」「詳細設計書」「実装計画書」が子ページとして自動生成・保存されます。これにより、「なぜそのコードになったのか」という設計の意図がドキュメントとしてチームに資産化されます。

### 2.2. GitLab：ソースコードと統合の現場（The Workshop）

GitLabは、純粋な「コードホスティング」および「継続的インテグレーション（CI/CD）」のプラットフォームとして割り切って使用します。

*   **Merge Request (MR) の管理**
    *   **運用:** HOKUSAIが実装したコードは、GitLab上にMRとして自動提出されます。このMRのURLは、NotionのTaskページにも自動的に書き戻されます。
*   **コード品質の担保と対話型レビュー**
    *   **運用:** MR提出後、GitLab CI/CDによる自動テストが実行されます。テスト結果や、人間（レビュアー）がMR上で行ったコメント（例: 「ここのエラーハンドリングを強化して」）は、HOKUSAIが読み取り、自律的にコードを修正して再提出します。
    *   **メリット:** コードに関する議論はGitLabに、仕様に関する情報はNotionに、と明確に分離されます。

## 3. 理想的な開発フロー（HOKUSAI導入後）

導入後の日々の開発サイクルは以下のようになります。

1.  **【企画・設計】** PM/リードが **Notion** 上でEpicを定義し、具体的なTask（チケット）を起票する。
2.  **【開発着手】** 開発者がローカル環境で `hokusai start <Notion Task URL>` を実行する。
3.  **【AI自律稼働】** HOKUSAIが要件を読み取り、調査・設計・計画を行い、結果を **Notion** に自動保存する。
4.  **【実装・提出】** HOKUSAIがコードを実装し、完了後に **GitLab** へ自動でMerge Request (MR) を作成する。
5.  **【レビュー】** チームメンバーが **GitLab MR** 上でコードをレビューし、必要に応じてコメントを残す（HOKUSAIがコメント対応を自動で行う）。
6.  **【マージ・完了】** MRがマージされると、HOKUSAIが最終的な変更サマリーを **Notion** に記録し、タスクが完了（DONE）となる。

---

## 4. 期待される導入効果

この役割分担を徹底することで、以下の効果が期待できます。

*   **Single Source of Truth（信頼できる唯一の情報源）の確立:**
    「仕様や進捗を知りたければNotionを見る」「最新のコードやレビュー履歴を知りたければGitLabを見る」という明確なルールが定着し、チーム内のコミュニケーションコストが激減します。
*   **AIコンテキストの最適化:**
    Notionに情報が集約されることで、HOKUSAI（AI）がプロジェクトの背景や関連仕様を正確に読み取りやすくなり、手戻りの少ない高品質なコード生成が可能になります。
*   **脱・属人化（ドキュメントの自動生成）:**
    これまで開発者の頭の中にしかなかった「設計の意図」や「調査結果」が、Notionにドキュメントとして自動的に残るため、後からの保守や引き継ぎが極めて容易になります。
