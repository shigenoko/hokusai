# HOKUSAI: Figma / Miro 連携要件定義書

## 1. 目的と背景

HOKUSAIは現在、Notion（ビジネス・要件）とGitLab（エンジニアリング・実行）を繋ぐオーケストレーションツールとして機能している。本要件は、ここに**Figma（デザイン）**と**Miro（企画・要件定義）**を統合し、ビジネス・デザイン・開発の3領域をHOKUSAIをハブとしてシームレスに連携させるための機能拡張を定義する。

### 期待される効果
1.  **デザイン意図の正確な実装**: Figmaから正確なデザインスペックを抽出し、AIによる実装の精度を向上させる。
2.  **企画段階からのシームレスな移行**: Miroのブレインストーミングやラフスケッチを、構造化された仕様やFigmaの下書きに自動変換する。
3.  **情報分断の解消**: Notion、GitLab、Figma、Miro間で「最新の情報がどこにあるか」「言った言わない」の問題をAIが仲介することで解決する。

---

## 2. システム・アーキテクチャ

HOKUSAIの拡張方針に基づき、以下のコンポーネントを追加・連携する。

### 2.1. インテグレーション追加
-   `hokusai/integrations/figma.py`: Figma REST APIクライアント
-   `hokusai/integrations/miro.py`: Miro API / MCP クライアント

### 2.2. プロンプトとノードの拡張
-   **Phase 2 (Research) / Phase 3 (Design)**: タスクに紐づくMiro/Figma URLを検知し、情報を取得・解析するロジックの追加。
-   **Phase 5 (Implement)**: 取得したデザイン制約（Figmaトークン）を実装エージェント（Claude Code等）のコンテキストに注入。
-   **Phase 8 (PR Draft)**: GitLabのMRにFigmaのプレビューやMiroの仕様URLを自動添付。

---

## 3. Figma 連携要件

Figmaは「デザインと実装の橋渡し」として機能する。

### 3.1. 取得する情報（Figma ➔ HOKUSAI）
AIが「見たまま」の実装を行うための構造化データ。
-   **スタイル定義**: カラーコード（HEX）、フォント情報、ドロップシャドウなどのエフェクト。
-   **レイアウトとDOM構造**: Auto Layout設定、要素の位置関係、コンポーネントの親子関係。
-   **アセット**: SVGアイコンデータ、画像書き出しURL。
-   **状態（バリアント）**: Hover、Disabledなどの状態定義。
-   **テキスト情報**: 実際のラベルテキストやデザイナーの指示コメント。

### 3.2. 書き込む情報（HOKUSAI ➔ Figma）
実装状況のフィードバック。
-   **ステータス同期**: 実装開始、PR作成完了などのステータス通知。
-   **AIからの質問**: 実装上の不明点をFigmaコメントとして投稿。
-   **プレビュー連携**: （将来構想）CIで生成された実装画面のスクリーンショットをFigmaに配置し、Visual Regression Testを支援。

---

## 4. Miro 連携要件

Miroは「ビジネスアイデアと要件定義の橋渡し」として機能する。

### 4.1. 取得する情報（Miro ➔ HOKUSAI）
非構造データからの要件抽出。
-   **テキスト・付箋**: ブレインストーミングの内容、要件の箇条書き。
-   **フロー・図解**: 図形とコネクタの関係性（ユーザーフローやアーキテクチャ図）。

### 4.2. 高度な変換フロー（Miro ➔ HOKUSAI ➔ Figma）
MiroのラフスケッチをFigmaの正規デザインへ変換する機能。
1.  **解読**: Miro上の図形配置から「ログイン画面のワイヤーフレーム」などの意図を解釈。
2.  **変換**: 意図に基づき、プロジェクトの正規デザインルール（8pxグリッドなど）を適用。
3.  **生成**: Figma APIを呼び出し、正規コンポーネントを用いた下書きフレームをFigma上に自動作成。

---

## 5. 想定されるユースケース（ユーザーフロー）

1.  **【企画】** PMがMiroで要件とラフなUIを検討する。
2.  **【起票】** NotionのタスクにMiroのURLを貼る。
3.  **【変換】** HOKUSAIがMiroを読み取り、FigmaにUIの下書き（正規コンポーネントベース）を自動生成する。
4.  **【デザイン】** デザイナーがFigmaで下書きをベースにデザインを完成させる。NotionタスクにFigma URLを追記する。
5.  **【実装開始】** `hokusai start` 実行。
6.  **【自律実装】** HOKUSAIがFigmaのデザインスペックを読み取り、AI（Claude Code等）に渡して正確にコーディングさせる。
7.  **【レビュー】** GitLabにMRが作成され、概要欄にFigmaのデザインプレビューが添付される。PMとエンジニアがMR上で確認・承認する。

---

## 6. 前提・制約事項

-   Miro連携は、公式またはコミュニティ提供の **Miro MCPサーバー** を活用することを第一候補とする（自然言語による操作の親和性が高いため）。
-   Figma連携は、**Figma REST API** を使用し、対象ファイルへのRead権限（必要に応じてWrite権限）を持つアクセストークンをHOKUSAIに設定する必要がある。
-   情報の正本（Source of Truth）の原則として、要件の正本は「Notion」、デザインの正本は「Figma」、コードの正本は「GitLab」とし、Miroは企画段階の一時的なキャンバス、または全体鳥瞰図として位置づける。
