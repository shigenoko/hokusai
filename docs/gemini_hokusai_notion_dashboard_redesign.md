# HOKUSAI ダッシュボード再定義設計書: Notion-Centric Architecture

**作成者:** Gemini CLI
**対象:** プロジェクト責任者、テックリード、システムアーキテクト
**目的:** Notionを主要コンテンツエリアおよびメインダッシュボードとし、HOKUSAI Web Dashboardを開発者向けオペレーションコンソールへと再定義する。

---

## 1. 背景とビジョン

ビジネスサイド（営業、マーケ、マネジメント）とエンジニアサイドの境界をなくすため、情報の正本を Notion に集約する。HOKUSAI の役割を「独立したツール」から「Notion を拡張する自律型エンジン」へと進化させる。

- **Notion:** 「何をするか（要件）」と「どうなったか（進捗）」を全員が見るための**メインダッシュボード**。
- **HOKUSAI Web Dashboard:** 「今、AIが具体的に何をしているか」をエンジニアが監視・デバッグするための**技術コンソール**。

---

## 2. システム構成の再定義

### 2.1. LangGraph の役割
LangGraph は HOKUSAI の「自律的な思考プロセス」を司るエンジンである。
- **役割:** 10のフェーズ間の遷移、条件分岐（エラー時のリトライ、人間への確認要求）、状態管理。
- **制御の移行:** LangGraph 自体を削除するのではなく、**LangGraph の「起動・停止・入力」のインターフェースを Notion に公開**する。

### 2.2. ダッシュボードの役割分担

| 機能 | Notion (メインダッシュボード) | HOKUSAI Web Dashboard (開発コンソール) |
| :--- | :--- | :--- |
| **主な対象者** | ビジネスサイド、PM、全ステークホルダー | エンジニア、テックリード |
| **表示情報** | Epic/タスク一覧、現在のフェーズ、成果物(設計書等)、ビジネス価値 | リアルタイム実行ログ、詳細なコード差分(Diff)、LLMトークン消費、詳細エラーログ |
| **操作** | タスクの起票、承認ボタン、フィードバック入力 | プロセスの強制停止、詳細設定の変更、デバッグ、再実行 |

---

## 3. 具体的な設計案

### 3.1. Notion メインダッシュボード (Project Hub)
Notion データベースのプロパティとオートメーションを活用し、HOKUSAI を制御する。

1.  **ステータス連動プロパティ:**
    - `HOKUSAI Status`: [Idle, Running, Waiting for Human, Completed, Failed]
    - `Current Phase`: [1. Prepare, 2. Research, ... 10. Record]
    - `HOKUSAI Control`: [Start, Pause, Resume, Abort] (Notion ボタン機能)
2.  **進捗可視化 (Progress Bar):**
    - 10フェーズの進捗率をプロパティで計算し、ビジネスサイドが直感的に進捗を把握できる。
3.  **成果物の自動集約:**
    - 各フェーズで生成された Notion 子ページへのリンクがメインページに自動で集約される。

### 3.2. HOKUSAI Web Dashboard (Dev Ops Console)
開発者が必要な「実行時の詳細」に特化させる。

1.  **ストリーミングログ:** Claude Code や git コマンドの生出力をリアルタイム表示。
2.  **インタラクティブ Diff:** 実装フェーズでの変更箇所をリッチな UI で確認。
3.  **フェーズグラフの可視化:** LangGraph のノード遷移をビジュアル化し、どこで停止しているかを把握。

---

## 4. 実装戦略: Notion からの制御

LangGraph を「外す」のではなく、Notion を「リモコン」にする仕組み。

### 4.1. Notion -> HOKUSAI (Trigger)
- **Notion ボタン / ステータス変更:** 
    - Notion のオートメーション機能、または外部連携ツール（Make, Zapier 等）を経由して HOKUSAI の API エンドポイントを叩く。
    - HOKUSAI サーバーはリクエストを受け取り、指定された Task URL で LangGraph ワークフローを起動する。

### 4.2. HOKUSAI -> Notion (Sync)
- **Phase 実行毎の同期:**
    - 各フェーズの開始・終了時に、HOKUSAI が Notion MCP または API を使用して、親タスクページの `Current Phase` や `Status` を更新する。
    - `waiting_for_human` 状態になった場合、Notion 上の「承認ボタン」が有効化される。

---

## 5. 移行のメリット

1.  **情報の非対称性の解消:** 営業担当者が GitLab やターミナルを見ることなく、Notion だけで最新の「開発の意図と進捗」を確認できる。
2.  **エンジニアの集中力維持:** エンジニアは Notion 上の議論を確認しつつ、詳細な技術問題は Web Dashboard でデバッグするという、最適な「情報の棲み分け」ができる。
3.  **一貫したユーザー体験:** ユーザー（社員）にとっては「Notion が仕事の場」となり、HOKUSAI はその裏で動く強力なアシスタントとして認識される。

---

## 6. 今後の課題

- **認証とセキュリティ:** Notion から HOKUSAI サーバーを安全に呼び出すための認証機構。
- **リアルタイム性の確保:** Notion の API 制限を考慮した効率的なポーリング/プッシュ通知の設計。
- **エラーハンドリング:** HOKUSAI 自体がクラッシュした場合の Notion 側への通知方法。
